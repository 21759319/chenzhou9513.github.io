Eureka

SpringBoot 中 Eureka 框架简介

框架

Eureka Server 服务注册中心

Application Service 服务提供者

Application Client 服务调用者

  • Eureka Server提供服务发现的能力，各个微服务启动时，会通过Eureka Client向Eureka Server进行注册自己的信息（例如网络信息），Eureka Server会存储该服务的信息；
  • 微服务启动后，会周期性地向Eureka Server发送心跳（默认周期为30秒）以续约自己的信息。如果Eureka Server在一定时间内没有接收到某个微服务节点的心跳，Eureka Server将会注销该微服务节点（默认90秒）；
  • 每个Eureka Server同时也是Eureka Client，多个Eureka Server之间通过复制的方式完成服务注册表的同步；
  • Eureka Client会缓存Eureka Server中的信息。即使所有的Eureka Server节点都宕掉，服务消费者依然可以使用缓存中的信息找到服务提供者。

CAP定理

具体案例 ： http://www.ruanyifeng.com/blog/2018/07/cap.html

数据一致性 Consistency

论文定义: Every read receives the most recent write or an error

向系统发起请求，读取系统中的某个值，必须是获取到的是最新值或者是返回error。

服务可用性 Avaliability

论文定义: Every request receives a (non-error) response – without guarantee that it contains the most recent write

向系统发起请求，不管返回的是不是最新值，必须保证系统能够在一定时间内响应(不能是error)。

分区容错性 Partition tolerance

论文定义: The system continues to operate despite an arbitrary number of messages being dropped (or delayed) by the network between nodes

无论有多少分区与分区之间的消息因为网络原因丢失了，系统依然会持续运行。

CAP三者不可能同时成立

因为CAP定理是针对分布式系统的，分布式系统是前提，因为网络总会有延迟或者故障的情况，CAP中的P总是要求成立的，分布式系统中不可能因为节点间网络故障而导致整个系统奔溃，所以我们只需要在C和A中做抉择。

为什么不可能同时成立:

假设以满足分区容错为前提，那么就有可能出现，网络中仅有两节点x,y, 且x与y之前网络断开的情况,假设节点x存的是最新数据，节点y还未和节点x同步

  1. 如果满足了C， 那么就必须等到网络恢复了才能响应服务，因为y必须和x同步才能保证一致性，A就不能保证
  2. 如果满足了A， 那么响应的时候就有可能访问到节点y(因为网络断开了,并不知道x,y谁是最新值)，那么不满足C

一般情况下，是保证AP，放弃C

不是完全放弃C， 就像区块链系统中一样，放弃C只是放弃强一致性。不做到实时保证一致性，而是保证最终一致性，即保证一段时间后系统的一致性。

Zookeeper 和 Eureka

Zookeeper 保证CP

因为zookeeper只有一个leader节点负责数据的操作，是能保证数据一致性的，如果leader宕机了，根据fast 算法选出新的leader， 但是由于要做主从节点间的同步， 在一段时间内A就无法保证了。

Eureka 保证AP

不具备主从关系，所有节点都是平等的。只保证弱一致性，当server 收不到某个client的心跳时，任然保存着这个client注册过的服务，这个server依然可以注册和查询服务，只有当client恢复正常后，再做同步。

Zookeeper 单语言， 以客户端形式访问

Eureka 多语言， 因为是以http形式访问

Eureka 自动保护

1.心跳

一般情况下，服务在Eureka上注册后，会每30秒发送心跳包，Eureka通过心跳包来判断服务是否健康，同时会定期删除超过90s没有发送心跳包的服务

2. 启动自我保护模式

收不到服务的原因

a. 服务主机自身的原因，比如程序停了，关机了。

b. 网络阻塞，Eureka server为了安全起见不会删除该服务

​ 如何判断是否网络阻塞，用eureka收不到服务的多少来判断，如果大批量收不到服务，服务的主机出问题了，即情况a，所以eureka设置了一个阈值， 15分钟内小于85%的服务心跳停止的话，那么可能只是网络的问题(有那么一小部分的心跳发送的时候阻塞了)，就启动自我保护模式，即不删除停止心跳的服务。

3. 为什么这么做

保持了CAP中的A

因为Eureka有对服务的返回值有缓存功能，所以即使服务心跳没了，也可以在保护模式下继续响应请求。
